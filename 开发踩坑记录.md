# iOS 越狱插件开发踩坑记录（GumJS WebSocket 实战）

> 本文记录把一个 Android Frida-GumJS 注入插件移植成 iOS Tweak 的全过程踩坑，
> 适用环境：**rootless 越狱**（iPad 6 / iPadOS 17.6，A10，无 PAC），Theos + GitHub Actions 编译。
> 每个坑按 **现象 → 原因 → 解决** 记录，方便以后写新插件直接抄。

---

## 0. 整体架构（先看这个，能避开 80% 的坑）

```
┌─────────────────────────────────────────────┐
│ 目标 App 进程                                  │
│  ┌──────────────┐   dlopen    ┌────────────┐ │
│  │ Tweak (.dylib)│ ─────────► │ 引擎 .dylib │ │
│  │  轻量, 只判断  │            │ Frida-GumJS │ │
│  │  + 加载引擎    │            │ + 自带通讯   │ │
│  └──────────────┘            └─────┬──────┘ │
└────────────────────────────────────┼─────────┘
                                      │ ws:// (纯 BSD socket)
                                   ┌──┴──┐
                                   │ PC  │ server.py 热重载脚本
                                   └─────┘
```

**关键设计决策（都是踩坑换来的）：**
1. **Tweak 与重引擎分离**：Tweak 只做"要不要注入"的判断，真正的 Frida 引擎做成独立 `.dylib`，命中目标 App 才 `dlopen`。避免每个进程都加载几 MB 的引擎。
2. **引擎只导出 2 个符号**：`gjws_start` / `gjws_cleanup`，其余全部隐藏（见坑 #3）。
3. **通讯用自带的纯 socket WebSocket**，不依赖系统网络栈（见坑 #7）。

## 1. 环境与工具链

### 坑 1.1：SDK 不存在
**现象：** `Error: Your chosen SDK, "iPhoneOS14.5.sdk", does not appear to exist.`
**原因：** Theos 默认找某个版本 SDK，但本地没装。
**解决：** 拉官方 SDK 仓库塞进 `$THEOS/sdks/`：
```bash
wget https://codeload.github.com/theos/sdks/zip/refs/heads/master -O /tmp/sdks.zip
unzip /tmp/sdks.zip -d /tmp/
cp -r /tmp/sdks-master/*.sdk $THEOS/sdks/
```

### 坑 1.2：rootless 路径
**现象：** 装上去插件不生效 / 找不到文件。
**原因：** rootless 越狱所有东西在 `/var/jb/` 下，不是传统 `/`。
**解决：**
- Makefile：`THEOS_PACKAGE_SCHEME = rootless`（并 `export`）。
- 代码里写死路径要用 `/var/jb/usr/lib/...`。
- `TARGET := iphone:clang:latest:14.0`（部署目标别写太低，否则 arm64e 报警告）。

### 坑 1.3：本地 Windows 没法编译
**结论：** Windows 开发，**用 GitHub Actions 编译最省事**（免维护 Linux+Theos 环境）。远程 Ubuntu 也行但要自己装工具链。最终方案：push 到 main → Actions 自动编译 → 产物发布到 gh-pages 当 Sileo 源。

## 2. Frida DevKit 集成与链接

### 坑 2.1：GumJS 嵌入式初始化
embedded GumJS（不是 frida-server）的标准启动姿势：
```c
gum_init_embedded();
GumScriptBackend *backend = gum_script_backend_obtain_qjs();  // QuickJS 后端
// ... 创建 script, set_message_handler, load ...
```
配合一个 `GMainLoop` 跑事件循环（脚本回调、定时器都靠它）。

### 坑 2.2：`.mm` 里 include frida 头报 module 错误
**解决：** 编译参数加 `-fno-modules -fno-cxx-modules -Wno-module-import-in-extern-c`，并用 pragma 包住：
```objc
#pragma clang diagnostic push
#pragma clang diagnostic ignored "-Wmodule-import-in-extern-c"
#include "frida-gumjs.h"
#pragma clang diagnostic pop
```

## 3. 【最大的坑】符号可见性导致 dlopen 崩溃

**现象：** 目标 App 一注入就 `Corpse failure`，秒崩。但用 `frida -f -F` 注入同一个 App 完全正常。日志显示崩在 `dlopen` 引擎那一刻（`before dlopen` 有日志，`after dlopen` 没有）。

**排查过程（方法论很重要）：**
1. 加 native 文件日志（`/var/tmp/gjws.log`）逐行打点 → 定位崩在 `dlopen`。
2. CI 里加诊断步骤，`dpkg-deb -x` 解出 `.dylib`，跑 `otool -L / -l / -hv` 和 `nm -gU`。
3. 发现引擎 `.dylib` **导出了几十万个符号**（V8、GLib、QuickJS 全暴露了），Theos 默认 `-undefined dynamic_lookup` 把所有符号都导出。
4. 这些符号和进程里别的库冲突，`dyld` 加载时崩。

**根因：** 静态链接了带 V8 的 frida-gumjs，默认全符号导出 → dyld 符号冲突。

**解决（决定性修复）：**
```makefile
libGJWSEngine_CFLAGS  = ... -fvisibility=hidden
libGJWSEngine_CCFLAGS = ... -fvisibility=hidden -fvisibility-inlines-hidden
libGJWSEngine_LDFLAGS = lib/libfrida-gumjs.a -lresolv \
    -Wl,-exported_symbols_list,Sources/engine_exports.txt
```
`Sources/engine_exports.txt`（只导出这两个）：
```
_gjws_start
_gjws_cleanup
```
导出函数加 `__attribute__((visibility("default")))`。
**验证：** `nm -gU` 导出符号从几十万降到 **2** 个，崩溃消失。

> 教训：**任何静态链接大型库（V8/JSC/protobuf…）做成 dylib 注入，都必须用 `exported_symbols_list` 白名单导出符号**，否则 dyld 冲突几乎必崩。这也是为什么 frida 官方 gadget 这么做。

## 4. 引擎与 Tweak 解耦

### 坑 4.1：dlopen 标志
**解决：** 用 `RTLD_LAZY | RTLD_LOCAL`（不要 `RTLD_GLOBAL`，会把引擎符号灌进全局命名空间，加剧冲突）。
```objc
void *handle = dlopen("/var/jb/usr/lib/libGJWSEngine.dylib", RTLD_LAZY | RTLD_LOCAL);
void (*start)(const char*) = dlsym(handle, "gjws_start");
```

### 坑 4.2：注入判断尽量轻
Tweak 的 `%ctor` 里只做：读配置 plist → 判断当前 bundleID 是否目标 → 是才 dlopen。别在 ctor 里做重活。

## 5. Settings / PreferenceBundle

### 坑 5.1：装完插件后，其它插件的设置项全消失！
**现象：** 安装本插件后，Settings.app 里所有 Tweak 的偏好面板都不见了。
**原因：** PreferenceBundle 的 Info.plist / 入口控制器有问题，导致 PreferenceLoader 加载链崩溃，连累其它面板。
**解决：**
- PreferenceBundle 的 `Depends` 写全：`preferenceloader, com.opa334.altlist`。
- 入口控制器继承正确的基类，别在 `specifiers` 里抛异常。
- 测试方法：`killall -9 Preferences` 后重进，看面板是否恢复。

### 坑 5.2：应用列表不全
**现象：** 自己实现的应用选择列表只显示"已签名安装的 App"。
**解决：** 用 **AltList**（`com.opa334.altlist`），继承 `ATLApplicationListSubcontrollerController`，用自定义谓词显示全部非 Apple App：
```objc
NSPredicate *predicate = [NSPredicate predicateWithFormat:
    @"NOT (atl_bundleIdentifier BEGINSWITH 'com.apple') AND atl_isHidden == NO"];
```
CI 里用 AltList 的 **`.tbd` stub + 拷头文件**，别去完整 build 它（会因 SDK 版本报错）。

## 6. CI/CD 自动编译 + Sileo 源

### 坑 6.1：GitHub Pages 当 Sileo 源
**方案：** Actions 编译出 `.deb` → 生成 `Packages` / `Packages.gz` / `Release` → push 到 `gh-pages` 分支。源地址即 `https://<user>.github.io/<repo>/`。
```bash
dpkg-scanpackages debs /dev/null > Packages
gzip -9c Packages > Packages.gz
# 别忘了 touch .nojekyll, 否则 Pages 会忽略某些文件
```

### 坑 6.2：卸载要干净
`control` 的 `Depends` 别乱加；配置 plist 放在固定路径，卸载脚本（`postrm`）里 `rm -f` 一并清理。

### 坑 6.3：【血泪坑·最坑】Sileo 源 URL 含下划线 `_` → 能刷新、但 deb 永远下不下来
**现象：** 源添加成功、能刷新、Sileo 里能看到包和版本号，但一点安装就报
`E: Can't find a source to download version 'x.x.x' of 'pkg:iphoneos-arm64'`。
**排查的弯路（教训）：** 一度以为是网络、缓存、`.bz2` 缺失、依赖、甚至本地同版本冲突，全是错的方向。**关键的排错动作**是「换一个完全不同的 CDN（github.io → jsDelivr）后报错一模一样」——这一步直接排除了托管/网络/格式，把矛头指向 URL 本身。
**根因：** Sileo 的已知 bug（[Sileo/Sileo#349](https://github.com/Sileo/Sileo/issues/349)）——**只要源 URL 路径里含下划线 `_`，Sileo 就无法从该源下载任何 deb**（能读索引，不能下载）。我们仓库名 `gumjs_ios_websocket` 带下划线，所以 `github.io` 和 `jsDelivr` 两个地址都中招。
**解决：** **仓库名 / 源路径绝对不要用下划线，用连字符 `-`**。把仓库重命名 `gumjs_ios_websocket` → `gumjs-ios-websocket`，源地址变 `https://用户名.github.io/gumjs-ios-websocket/`，立刻正常。
> 反证：alias20 的源 `alias20.gitlab.io/apt/` 路径里没有下划线，所以一直好用。
> **铁律：iOS 源相关的用户名、仓库名、路径，全程只用小写字母/数字/连字符，永不用下划线。**

### 坑 6.4：Sileo 报错笼统 → 用「换 CDN」和「命令行 apt」定位
- **"Can't find a source to download"**：先怀疑 URL 含下划线（坑 6.3）；再用命令行 `apt update` / `apt install --reinstall xxx` 看**真实**错误（Sileo GUI 会把详细错误吞掉）。
- **排除托管/网络的快招**：把同样的内容换一个 CDN 加成源，若报错完全相同，则问题在源内容/URL，不在网络。
- **验证源是否健康**（在电脑上 curl，不用上设备）：
```bash
curl -sI  https://<源>/Packages          # 200?
curl -s   https://<源>/Packages | grep Version
curl -sI  https://<源>/debs/xxx.deb       # 200? Content-Length 对不对?
```

### 坑 6.5：压缩索引优先给 `Packages.bz2`
Sileo 生态按 `.zst→.xz→.bz2→.gz→明文` 探测，**`.bz2` 最主流**。只给 `.gz` 时，部分 Sileo 探测 `.bz2` 拿到 404 后行为异常。`build_repo.sh` 里同时生成：
```bash
dpkg-scanpackages debs /dev/null > Packages
bzip2 -9kc Packages > Packages.bz2
gzip  -9c  Packages > Packages.gz
```
（注：GitHub Pages 对 `.gz` 不会加 `Content-Encoding: gzip`，所以 `.gz` 本身没坑，但 `.bz2` 一定要补上。）

### 坑 6.6：GitHub Pages 不支持目录浏览（`debs/` 报 404 是正常的）
访问 `https://<源>/debs/` 会 404，因为没有目录索引页。这**不代表源坏了**——Sileo 只读 `Packages` 拿 `Filename` 再下具体文件，从不列目录。验证要用**完整文件 URL**，别用目录 URL。

### 坑 6.7：国内访问 github.io 不稳 → jsDelivr 镜像 / 命令行装
- jsDelivr 可直接镜像 gh-pages 当源：`https://cdn.jsdelivr.net/gh/<用户>/<仓库>@gh-pages/`（同样**别带下划线**）。
- jsDelivr 对分支引用有缓存，发新版后用 purge 刷新：`curl https://purge.jsdelivr.net/gh/<用户>/<仓库>@gh-pages/Packages`。
- 实在不行就手动 `curl -L -o /tmp/x.deb <deb地址> && dpkg -i /tmp/x.deb`（`dpkg -i` 不走 APT 源解析，最稳）。

### 坑 6.8：安装/卸载会自己强制重启 SpringBoard（maintainer scripts）
**现象：** 包一装/一卸就自动 respring，而规范 tweak 是 Sileo 弹按钮、用户手动点才重启。
**原因：** `layout/DEBIAN/postinst`、`postrm` 里硬写了 `sbreload || killall -9 SpringBoard`（`prerm` 里 `killall -9 Preferences`）。
**解决：** **删掉 maintainer scripts 里的强制 `killall`/`sbreload`**，只保留 `rm` 清理。Sileo 检测到安装了 Substrate tweak（`DynamicLibraries/*.dylib`+plist）会自动在完成界面给「重启 SpringBoard」按钮，由用户手动点。
> 注：注入目标 App 的 tweak 其实不需要重启 SpringBoard 就生效（目标 App 下次启动即加载），重启主要为刷新偏好面板。别替用户做重启决定。

## 7. 通讯层：别 hook 到自己

### 坑 7.1：插件通讯被用户脚本 hook 了
**现象：** 写脚本 hook `NSURLSession`（尤其 `-[NSURLSessionTask resume]`）时，App 必崩。
**原因：** 插件自己用 `NSURLSessionWebSocketTask` 回传日志。用户 hook `NSURLSession` 基类方法时，**连插件自己的通讯 task 一起 hook 了** → 在通讯线程里触发用户回调 → 里面再 `console.log`（又走通讯）→ **递归/重入崩溃**。
**解决（彻底隔离）：** 把通讯层从 `NSURLSession` 换成**自带的纯 BSD socket WebSocket 客户端**（自己实现 RFC6455 握手 + 帧编解码，只用 `socket/connect/send/recv/close`）。这样用户 hook 任何高层网络库（`NSURLSession`/`CFNetwork`/`SSL`）都碰不到插件通道。

> 通用原则：**注入框架的自身通讯，要走用户最不可能去 hook 的底层原语**，和被分析的目标彻底隔离。代价是要自己实现协议，但换来稳定。

## 8. 【GumJS 脚本】embedded 与 frida-server 的差异（写脚本最容易踩）

> 我们用的是嵌入式 GumJS（QuickJS 后端），**很多 frida-server 下能用的 API 在这里不可用或会崩**。

### 坑 8.1：`ObjC` 全局对象不存在
**现象：** `ReferenceError: 'ObjC' is not defined`。
**原因：** embedded GumJS 不带 ObjC 桥。
**解决：** 用底层 runtime 自己拼：
```js
var objc_msgSend = new NativeFunction(Module.findGlobalExportByName('objc_msgSend'),
    'pointer', ['pointer', 'pointer']);
var sel_registerName = new NativeFunction(Module.findGlobalExportByName('sel_registerName'),
    'pointer', ['pointer']);
function SEL(n){ return sel_registerName(Memory.allocUtf8String(n)); }
function call(o,n){ return objc_msgSend(o, SEL(n)); }
```

### 坑 8.2：Frida 17 找导出函数的 API 变了
**解决：** 用 `Module.findGlobalExportByName('xxx')`（旧的 `Module.findExportByName(null,'xxx')` 在新版可能不灵）。

### 坑 8.3：【重要】`ApiResolver('objc').enumerateMatches()` 在 embedded gum 里崩
**现象：** `new ApiResolver('objc')` 能创建，但一调 `enumerateMatches('-[SomeClass method:]')` 就秒崩，且 `console.log` 都来不及发出。
**排查：** 用 native 文件日志逐行打点，确认崩在 `enumerateMatches`。
**解决：** **完全不用 ApiResolver**，改用 ObjC runtime C 函数直接拿方法地址：
```js
var objc_getClass = new NativeFunction(Module.findGlobalExportByName('objc_getClass'),
    'pointer', ['pointer']);
var class_getInstanceMethod = new NativeFunction(
    Module.findGlobalExportByName('class_getInstanceMethod'), 'pointer', ['pointer','pointer']);
var method_getImplementation = new NativeFunction(
    Module.findGlobalExportByName('method_getImplementation'), 'pointer', ['pointer']);

function hookMethod(clsName, selName, onEnterFn){
    var cls = objc_getClass(Memory.allocUtf8String(clsName));
    if (cls.isNull()) return false;
    var m = class_getInstanceMethod(cls, SEL(selName));
    if (m.isNull()) return false;
    var imp = method_getImplementation(m);
    Interceptor.attach(imp, { onEnter: onEnterFn });   // 直接 attach IMP, 稳
    return true;
}
```
> 经验：embedded gum 里要 hook ObjC 方法，**走 `objc_getClass → class_getInstanceMethod → method_getImplementation → Interceptor.attach(imp)`**，别用 `ApiResolver`。

### 坑 8.4：抓 HTTP/2 响应（拦 block 的 invoke）
SSL 层只能抓 HTTP/1.1 明文（HTTP/2 是 HPACK 二进制）。要抓 HTTP/2 解密后的响应，在 `dataTaskWithRequest:completionHandler:` 的 `onEnter` 里拿到 completionHandler block，读它的 invoke 指针再 attach：
```js
var invoke = block.add(16).readPointer();   // arm64 Block_layout: invoke 在偏移 +16
Interceptor.attach(invoke, { onEnter: function(args){
    // args[0]=block, args[1]=NSData*, args[2]=NSURLResponse*, args[3]=NSError*
}});
```
completionHandler 的参数位置 = selector 冒号数 + 1（`args[0]=self, args[1]=cmd`）。

### 坑 8.5：读 NSData / 容错文本
对可能含二进制的 buffer，别用 `readUtf8String`（遇非法字节抛异常），用 `readByteArray` 手动转：
```js
function safeText(buf, len, max){
    var arr = new Uint8Array(buf.readByteArray(Math.min(len, max||8192)));
    var s=""; for (var i=0;i<arr.length;i++){ if(arr[i]===0) break; s+=String.fromCharCode(arr[i]); }
    return s;
}
```

## 9. PC 端 server.py

- **相对路径坑：** `os.path.dirname("relative.js")` 返回空串导致 `FileNotFoundError`。用 `os.path.abspath(sys.argv[1])`。
- **弃用警告：** `asyncio.get_event_loop()` 已弃用，用 `asyncio.run()`。
- **端口占用：** `OSError: [Errno 10048] address already in use` —— 上一个 python 进程没退，`taskkill` 掉对应 PID。

## 10. 调试方法论（最值钱的部分）

1. **native 文件日志是救命稻草。** App 崩太快时 `console.log` / `os_log` 都可能丢，**直接写文件并 `fclose`（强制落盘）** 才可靠：
   - C 侧：`GJWS_FLOG` 宏写 `/var/tmp/gjws.log`。
   - 脚本侧：用 `NativeFunction` 调 `fopen/fputs/fclose` 写 `/var/tmp/gjws_script.log`，**每个关键步骤前后各打一行**，崩了看最后一行就知道崩在哪。
2. **二分法定位。** 用顶部布尔开关逐个启用 hook 模块，缩小到具体哪一步。
3. **CI 里做二进制诊断。** `otool -L`（依赖）、`otool -hv`（header/命名空间）、`nm -gU | wc -l`（导出符号数）。符号数异常多 = 符号可见性没控制。
4. **对照法。** 同一脚本 `frida -f -F` 能跑、嵌入式崩 → 差异在 embedded gum 的 API 兼容性（如坑 8.3）。
